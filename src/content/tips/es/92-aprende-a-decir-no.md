DHH es famoso por decir no: **cada feature que no construyes es mantenimiento que no tienes**.

## El coste del "sí"

```markdown
"Sí, podemos añadir esa feature"

Implica:
- Código que escribir
- Tests que mantener
- Documentación que actualizar
- Bugs que arreglar
- Usuarios que soportar
- Complejidad que gestionar
```

## Alternativas al "no"

```markdown
- "No ahora, pero lo consideramos para Q3"
- "¿Qué problema estás intentando resolver?"
- "Podemos hacerlo así como alternativa"
```

## Reflexión final

El software más exitoso no es el que tiene más features. Es el que resuelve bien un problema específico.
